## What this app is
A private hunting **portfolio** — *Instagram × Strava, for hunting*. A user has a
profile and a collection of **kill records** (the core entity); each kill is one
card with the animal photo as cover. Hierarchy:

    Profile → Country → Year → Animal (kill card)

Two views of the same records:
- **Feed tab** — reverse-chronological by kill date (Instagram-style).
- **Group-by-location tab** — grouped by country / place.

Accounts run on Firebase Authentication and every portfolio is private to its
owner. Friend search / "see what your friends shot" comes later; any future
social view must be a deliberate read-only projection, never direct access to
another user's private source records.

## Stack (decided defaults — ask before replacing any of these)
- **Vite + React 19 + TypeScript + React Router 7**, deployed on **Vercel** as a
  clean-URL SPA.
- **Tailwind CSS + shadcn/ui** components; **react-hook-form** for forms.
- **Firebase**: Auth + Cloud Firestore + Cloud Storage. All documents and files
  are rooted at the authenticated UID and guarded by `firestore.rules` /
  `storage.rules` (tested via `npm run test:rules`).
- **MapLibre GL JS** for the satellite walk view; **GPX parsing** in `lib/gpx`
  for Garmin/Strava route import (store raw GPX verbatim, derive distance/time).
- **Capacitor 8** wraps the same web app for iOS/Android
  (`@capacitor-firebase/authentication` for native sign-in). Keep client code
  free of Node-only/server-only assumptions so the wrap stays cheap.

## Domain model
Authoritative definition: `vault/0-Overview/Data-Model.md`. Summary — a **Kill**
record holds:
- `species`, `coverPhoto`, `media[]` (photos + videos)
- `country`, `date` (→ `year` derived), exact `killTime` (captured when
  location is set), `location` (lat/lng + place name)
- `weapon`: `type` (`rifle` | `bow`) + exact model; rifle → `caliber`,
  bow → bow type
- `ammunition`: grain + brand/color detail
- `route`: raw imported GPX + derived `distanceKm` / `durationMin`, rendered on
  the satellite map
- `description`: free-text hunt narrative

## Critical invariants (non-negotiable)
Before touching any guardian file below, understand which invariant it protects.

1. **Kill-record integrity.** Core facts (species, date, `killTime`, `location`,
   `weapon`) are never silently lost or mutated on edit; media/route attachments
   stay linked to the correct record.
   *Guarded by:* `lib/domain/` + `lib/firebase/` persistence; every edit path.
2. **Factual data is authoritative, never fabricated.** `killTime`, `location`,
   caliber/grain, the GPX track and its derived distance/time are real-world
   facts (relevant to tags/seasons/legality). Derive metrics from source data;
   never invent or overwrite source values.
   *Guarded by:* `lib/gpx/`, location/time capture in the kill form.
3. **Media is never destroyed.** Uploaded photos/videos and original GPX files
   are preserved; deletes are explicit, user-confirmed actions.
   *Guarded by:* media handling in `lib/` + Storage layer.
4. **Multi-user isolation.** A user reads/writes only their own portfolio.
   *Guarded by:* `firestore.rules`, `storage.rules`, `lib/firebase/`, and the
   rules tests in `tests/rules/`. Any rules change must keep
   `npm run test:rules` passing and should get a security review.

## Gotchas (add a line here whenever a mistake happens twice)
- The GitHub Pages landing page deploys from branch
  `codex/firebase-hunting-portfolio-v1` `/docs` — **not** from `main`.
- The real brand logo SVG (`head-light.svg`) is >256KB and gets truncated by
  tools with size caps; the hand-authored antler emblem at
  `docs/logo/head-{amber,light}.svg` is the working logo. Never hand-copy
  base64 image data — it corrupts.
- `.env.local` holds real Firebase config (git-ignored); `.env.example` is the
  committed template. Never put a real key in a committed file.
- `google-services.json` must be regenerated from the Firebase console when
  OAuth clients change — don't hand-edit it.
